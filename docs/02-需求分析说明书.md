## 一、核心检测规则（MVP 10 条）

### 规则目录

```
src/rules/
├── no-duplicate.ts           # 字面重复检测
├── no-semantic-duplicate.ts  # 语义重复检测
├── no-conflict.ts            # 冲突指令检测
├── no-overconstrain.ts       # 过度约束检测
├── no-verbose.ts             # 冗余表述检测
├── no-global-path-rule.ts    # 误用全局规则检测
├── no-stale-reference.ts     # 死引用检测
├── no-null-effect.ts         # 无效约束检测
├── no-skill-bloat.ts         # Skill 体积膨胀检测
├── max-length.ts             # 规则总长度检测
├── no-overlap-skills.ts      # 跨 Skill 重叠检测
└── no-missing-frontmatter.ts # YAML 头缺失检测
```

### R1: no-duplicate（字面重复）

**检测**: 同一文本块在配置文件中出现 ≥2 次

```
❌ "使用 TypeScript 严格模式" 出现 2 次 (行 2, 行 4)
   → 浪费 ~12 tokens/次 × 每天 ~100 轮 = ~1,200 tokens/天
   → fix: 合并为 1 处
```

### R2: no-semantic-duplicate（语义重复）

**检测**: 两条规则用不同措辞表达同一含义

```
❌ "提交前运行 npm run test" (行 14) 与 "提交前运行 npm run test 并全部通过" (行 16)
   → 语义重叠 85%，冗余
   → fix: 合并为 "提交前运行 `npm run test`（禁止跳过）"
```

**实现**: MVP 阶段用基于关键字的相似度匹配，非 LLM。

### R3: no-conflict（冲突指令）

**检测**: 两条规则表达矛盾语义

```
❌ "始终使用单引号" (行 12) vs "使用双引号" (行 28)
   → 两条规则冲突
   → fix: 需要人工判断，标记为 WARNING 不自动修复
```

### R4: no-overconstrain（过度约束）

**检测**: 对不适用场景施加约束

```
⚠️ "所有函数必须添加 JSDoc 注释" 但项目不是 TypeScript
   → 对非 TS 项目的 JSDoc 约束无效，浪费 ~15 tokens/轮
```

### R5: no-verbose（冗余表述）

**检测**: 冗长措辞可简化

```
❌ "请务必确保一定在提交前运行所有测试用例" (25 tokens)
   → "提交前运行测试" (6 tokens) — 节省 76%
```

**实现**: 基于常见冗余模式字典（"请务必确保" → "确保"、"一定必须" → "必须"）。

### R6: no-global-path-rule（误用全局规则）

**检测**: path-scoped 规则写成全局规则

```
⚠️ "对于 docs/ 目录" → 写在全局 CLAUDE.md 而非 path-scoped rules
   → 每次对话都加载，但只在 docs/ 场景有效
   → 浪费 ~200 tokens/轮 × 非 docs 场景
```

### R7: no-stale-reference（死引用）

**检测**: 引用了不存在的文件/目录/Skill

```
❌ CLAUDE.md 引用了 "@docs/architecture/overview.md" → 文件不存在
   → Agent 尝试读取会失败，浪费 API 调用
```

### R8: no-null-effect（无效约束）

**检测**: 约束无实际效果

```
⚠️ "使用 TypeScript 严格模式" → 但项目无 tsconfig.json
   → 约束无法执行，浪费 tokens 且误导 Agent
```

### R9: no-skill-bloat（Skill 体积膨胀）

**检测**: SKILL.md 超过行数阈值

```
⚠️ imagegen-frontend-mobile/SKILL.md: 1465 行 (阈值: 500 行)
   → 每次触发此 Skill 消耗 ~8K input tokens
   → 建议拆分或精简
```

**阈值**: `--max-skill-lines` 参数，默认 500。

### R10: max-length（规则总长度膨胀）

**检测**: 配置文件总 token 数超过阈值 → AI 遵守率下降

```
⚠️ CLAUDE.md 总长度: 412 tokens (阈值: 300 tokens)
   → AI 遵守率风险: 400 行时从 94% 降至 71%
   → 建议精简规则
```

**阈值**: `--max-tokens` 参数，默认 300。

## 二、跨文件检测

### R11: no-overlap-skills（跨 Skill 重叠）

**检测**: 两个 Skill 的触发域或指令高度重叠

```
⚠️ design-taste-frontend/SKILL.md 与 design-taste-frontend-v1/SKILL.md
   → description 重叠度: 82%
   → 建议卸载 v1（旧版）或标记为 deprecated
```

### R12: no-missing-frontmatter（YAML 头缺失）

**检测**: SKILL.md 缺少 `---` YAML frontmatter 或缺少必要字段

```
❌ minimalist-ui/SKILL.md: 缺少 `name` 字段
   → Skill 可能无法被正确识别和加载
```

## 三、输出格式

### `ai-lint`（默认扫描）

```
❯ npx ai-lint

  CLAUDE.md        健康度: 62/100 ⚠️
  AGENTS.md        健康度: 78/100 ✅
  brandkit/SKILL   健康度: 55/100 ⚠️

  ── 3 个文件，3 个问题 ──

  CLAUDE.md
  ❌ [no-duplicate] "TypeScript 严格模式" 出现 2 次 (行 2, 行 4) — 浪费 ~1,200 tokens/天
  ❌ [no-verbose] "请务必确保一定在提交前运行所有测试用例" (行 14) — 可节省 76%

  brandkit/SKILL.md
  ⚠️ [no-skill-bloat] 798 行 (阈值: 500) — 每次触发消耗 ~5K input tokens

  ── 2 个修复建议 1 个警告 ──
  💡 运行 `ai-lint fix` 自动修复 2 个问题
```

### `ai-lint --cross-files`（跨文件检测）

```
❯ npx ai-lint --cross-files

  ⚠️ [no-overlap-skills] design-taste-frontend ↔ design-taste-frontend-v1
     description 重叠度: 82%
     → 建议卸载 design-taste-frontend-v1
```

## 四、CLI 命令

```
ai-lint                    # 扫描所有 AI 配置
ai-lint <file>             # 扫描指定文件
ai-lint --cross-files      # 启用跨文件检测
ai-lint --rules=<ids>      # 只运行指定规则
ai-lint fix                # 自动修复（跳过冲突类）
ai-lint fix --dry-run      # 预览修复效果
ai-lint explain <rule-id>  # 解释规则
ai-lint stats              # 健康度趋势
ai-lint init               # 生成规范模板
ai-lint --ci               # CI 模式（有问题 exit 1）
ai-lint --json             # JSON 输出（供工具消费）
```

## 六、验收标准

- [ ] `ai-lint` 扫描当前目录全部 AI 配置文件

- [ ] `ai-lint --cross-files` 检测出跨 Skill 重叠
- [ ] `ai-lint --ci` 有问题时 exit 1
- [ ] 扫描 15 个 Skill + CLAUDE.md + AGENTS.md ≤ 1 秒
- [ ] 零网络请求
